# frontdesk — AI assistant guidelines

This file provides guidance for any AI (or human) working on this repository.
The codebase is structured around a few load-bearing decisions; do not undo
them without a deliberate, documented reason.

## Repo layout

This directory groups two related projects:

- [backend/](backend/) — FastAPI + LangGraph backend with strict hexagonal DDD.
  See [backend/CLAUDE.md](backend/CLAUDE.md) for backend-specific rules.
- [frontend/](frontend/) — React 19 + Mantine 8 owner dashboard.
  See [frontend/CLAUDE.md](frontend/CLAUDE.md) for frontend-specific rules.

Each subproject owns its own dependencies, tests, and CLAUDE.md. **Always
read the subproject's CLAUDE.md before doing work inside it** — that file
is the source of truth for commands, conventions, and patterns.

## Knowledge base — read these before re-walking the code

The [docs/](docs/) folder contains durable mental-model documents. Treat
code as the source of truth; use docs as the map.

- [docs/architecture.md](docs/architecture.md) — Big picture, multi-tenant model, layer boundaries
- [docs/rag.md](docs/rag.md) — RAG ingestion + hybrid retrieval design
- [docs/escalation.md](docs/escalation.md) — Question lifecycle + state machine
- [docs/conversations.md](docs/conversations.md) — Persistence model + tiered tool compression

If a doc disagrees with the code, the code wins — update the doc in the
same change.

## Working across both projects

When a change spans backend and frontend (new endpoint + UI, schema rename, etc.):

1. Land the backend change first so the API contract is stable.
2. Mirror naming conventions across the boundary — backend uses snake_case
   in JSON, the frontend types follow the same shape.
3. Run each project's own verification block (lint / typecheck / tests)
   inside that project before declaring the change done.

## Architectural decisions (load-bearing)

These are deliberate choices, validated through real engineering trade-offs:

### 1. Strict hexagonal DDD with framework isolation

The domain layer has **zero** imports from `application`, `infrastructure`,
`drivers`, or `ai`. The application layer has **zero** imports from
`infrastructure` or `drivers`. LangChain and LangGraph live ONLY in
`infrastructure/llm/` and (when added) `infrastructure/ai/`, hidden behind
domain ports (`LLMClientPort`, `EmbeddingPort`, `RetrieverPort`, etc.).

The agent layer (when added) may import `langchain_core.messages` as a type
only — never `langgraph` directly. This makes the framework swappable and
the test surface small.

### 2. DB-as-truth for conversation persistence

LangGraph is used for per-turn orchestration only. The messages table is
the cross-turn source of truth — not `AsyncPostgresSaver`. This trades a
small amount of glue code for transparent queries, a readable admin UI,
and version-independent persistence.

### 3. Tiered tool compression with arguments preserved

Recent tool exchanges stay verbatim (`tool_use` / `tool_result` blocks the
LLM was trained on). Older exchanges roll up to a structured summary while
the original `tool_args` + `tool_result` are retained in JSONB for UUID-
driven recovery. This fixes the gap in naive "paraphrased system message"
patterns where the LLM loses awareness of what it previously asked.

### 4. Token + cost as a real ledger (not a callback)

Every LLM call writes a `token_usages` row with provider, model, tenant,
thread, request ID, plus per-segment cost in `Decimal(18, 8)`. Pricing
lives in `domain/llm_usage/pricing.py` so the cost math stays domain-side.
Aggregation happens in SQL, not in Python.

### 5. Hybrid retrieval with tenant isolation at every query

Vector (pgvector HNSW, cosine ops) + BM25 (Postgres tsvector + GIN) fused
via Reciprocal Rank Fusion (k=60). Every retrieval query filters
`tenant_id` — never trust callers to scope. Document chunks live in a
partitioned table; v1 is one user per tenant but the schema supports
many-to-many.

## Conventions

- **Commits**: conventional-commit format (`feat(backend): ...`,
  `fix(...)`, `chore(...)`, `docs(...)`). Body explains the *why*; the
  `git diff` shows the *what*.
- **Migrations**: always autogenerated via `alembic revision
  --autogenerate -m "..."` — never hand-written. Pgvector imports are
  added manually after generation if needed.
- **Tests**: backend has both `tests/unit/` and `tests/integration/`.
  Integration tests hit a real Postgres `frontdesk_test` database.
- **Secrets**: never commit. `.env.example` is the contract;
  `.env` is gitignored.

## CI checks (must pass before any commit)

Backend:
```bash
cd backend
uv run ruff check src/ tests/
uv run ruff format --check src/ tests/
uv run mypy src/                       # strict mode, must be 0 errors
uv run pytest tests/ --tb=short -q
```

Frontend:
```bash
cd frontend
npm run lint
npm run typecheck
npm test
```

## Things to NEVER do

- Never commit a `langchain_*` or `langgraph` import into `domain/`,
  `application/`, or the `ai/` layer (except for `langchain_core.messages`
  as a type, allowed in `ai/agents/`).
- Never hand-edit a migration to add table-creation logic — generate it,
  then add only the things autogenerate misses (e.g., extension
  CREATE EXTENSION, pgvector imports).
- Never raise `HTTPException` in a use case — use domain exceptions and
  let the error handler map them.
- Never trust client-provided `tenant_id`; resolve it from the
  authenticated user's `user_tenants` membership.
- Never commit without running the CI checks above.
