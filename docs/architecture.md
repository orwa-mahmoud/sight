# Architecture

frontdesk is a multi-tenant AI assistant that answers questions on its
owner's behalf. The architecture is shaped by three goals that often pull
in different directions and have to be reconciled:

- **Sound DDD**: domain logic stays pure and testable; framework choices
  are reversible without rewriting business rules.
- **Production retrieval quality**: hybrid search with tenant isolation
  enforced at every query, no cross-tenant leaks.
- **Honest cost accounting**: every LLM call accountable to a tenant +
  request + provider + model, with cost in `Decimal(18, 8)`.

Dependencies point **inward only**. The domain has zero imports from any
other layer.

## Multi-tenant model

Every domain entity that holds user data carries a `tenant_id` column.
Every repository query filters by it. There's no row-level security at the
Postgres layer in v1 (it would be a defense-in-depth addition for v2);
isolation is enforced in the application layer + repositories.

**Resolution rule:** never trust client-provided `tenant_id`. Routes
resolve it from `user_tenants` via the authenticated user's identity:

```python
async def _resolve_tenant_id(current_user, uow) -> UUID:
    links = await uow.user_tenants.list_for_user(current_user.id)
    if not links:
        raise AuthenticationError(...)
    return links[0].tenant_id
```

v1 enforces one tenant per user (1:1), but the schema supports many-to-many
via the `user_tenants` join table. v2 lets a user own multiple tenants and
adds team members per tenant with the `staff` role.

## Persistence

PostgreSQL 17 with pgvector for vector columns, native `tsvector` for BM25.
SQLAlchemy 2.0 with async + Alembic migrations (always autogenerated).

Key tables:

- `tenants`, `users`, `user_tenants` — multi-tenant primitives
- `conversations`, `messages` — threads with full tool exchange fidelity
  (tool_args + tool_result preserved in JSONB)
- `documents`, `chunks` — RAG corpus. `chunks.embedding` is `Vector(1536)`
  with an HNSW (cosine) index; `chunks.content_tsvector` is a generated
  column with a GIN index
- `token_usages` — append-only LLM call ledger
- `questions` — escalation state machine

## Request flow (representative)

A typical authenticated request:

1. Client hits `POST /api/v1/questions/{id}/reply` with bearer token
2. `get_current_user` decodes JWT, loads User from DB
3. `get_uow` opens a session + builds a UnitOfWork wrapping all repos
4. Route resolves `tenant_id` from `user_tenants`
5. Route instantiates `ReplyToQuestionUseCase(uow=uow)` and runs `execute`
6. Use case loads `Question` via the repo, calls `Question.resolve(...)`
   (domain method enforces state machine), saves back
7. On exit, `get_uow` commits; on exception, rolls back
8. Domain events from `Question` would fire here in Phase 8 (event bus
   for the channels-side relay)
